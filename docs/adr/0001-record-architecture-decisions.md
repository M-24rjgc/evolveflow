# ADR-0001：记录架构决策（ADR 约定）

- **状态**：已接受
- **日期**：2026-06-19

## 背景

EvolveFlow 是一个长期迭代的项目，且开发节奏是"隔很久回来看一次"。
如果没有决策记录，每次回来都会问"当初为什么这么选"，
要么重复纠结已经决定过的问题，要么误改掉有意的取舍。

## 决策

采用 ADR（Architecture Decision Record）记录所有有取舍的技术决策。

### 约定

- 存放位置：`docs/adr/`
- 命名：`NNNN-短标题-用连字符.md`（NNNN 从 0001 递增，不复用编号）
- 每条 ADR 包含固定小节：
  - **状态**：提案 / 已接受 / 已废弃 / 已被 NNNN 取代
  - **背景**：为什么需要做这个决策？面临什么约束？
  - **决策**：选了什么？怎么落地？
  - **后果**：带来什么好处、什么代价、什么风险？
  - **备选方案**：考虑过但没选的方案，以及为什么没选。
- ADR 一旦写定不可改（它是历史记录）；决策变更就新写一条，
  并在旧条标注"已被 ADR-NNNN 取代"。

### 何时写 ADR

- 选型决策（框架、库、外部服务、存储方案）。
- 架构层面的取舍（分层、进程模型、通信机制）。
- 影响多包或多人的约定（命名、目录结构、迁移策略）。
- 任何"以后可能被质疑为什么这么做"的决定。

### 何时不写

- 纯 bug 修复（写 commit message 即可）。
- 局部实现细节（写代码注释）。
- 没有真正取舍的决定（只有一个合理选项时）。

## 后果

- **好处**：决策有据可查，新人（含未来的自己）能快速理解"为什么"。
- **代价**：写 ADR 有额外开销；但只在重要决策时写，开销可控。
- **风险**：ADR 可能过时但未标注废弃——靠"决策变更即新写一条"的纪律缓解。
