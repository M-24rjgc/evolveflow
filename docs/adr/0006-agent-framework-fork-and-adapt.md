# ADR-0006：Agent 框架选型哲学——fork 改造，不重写，不追上游

- **状态**：已接受
- **日期**：2026-06-19
- **相关**：CONTEXT.md "愿景形态"、`.scratch/agent-framework-replacement/decision-map.md`

## 背景

EvolveFlow 当前的 AI runtime 是自研的（sidecar + 手写 loop/client/tools），
已暴露多个根本性问题（会话不持久化、1700 行上帝进程、压缩/重试 bug 频发）。
决定换用一个成熟的 Agent 框架作为底座。

在选型之前，需要先定下"如何使用框架"的哲学——这比"选哪个框架"更根本，
因为它决定了候选范围和评判标尺。

## 决策

采用 **F2：Fork 一次，自己接管** 的方式使用框架。

具体含义：

1. **复制改造，不重写**：把框架代码 fork 下来改，**不是**"学习框架思想后自己实现"。
   后者耗时、效果差、浪费生命。
2. **不要求功能完全匹配**：候选可以是企业级/多用户框架。选定期望后，
   砍掉不需要的部分（多用户、云同步、计费等），保留架构和核心能力。
3. **Fork 一次，不再追上游**：把框架当作起点，之后 EvolveFlow 按自身需求独立演化。
   不做长期 cherry-pick 上游更新（单人项目扛不住企业级框架的更新节奏）。
4. **保留 EvolveFlow 的核心资产**：domain/storage/capabilities 层是自研护城河，
   框架替换只动 runtime 层（AI 编排、会话、工具调度）。

## 后果

- **好处**：
  - 选型自由度高：架构好但社区小的框架也可入选（反正自己维护）。
  - 不受上游节奏绑架，完全按 A 层（自用优先）需求演化。
  - 避免重写的巨大浪费——直接站在成熟代码肩膀上。
  - 明确边界：只换 runtime，核心业务资产不动，风险可控。
- **代价 / 风险**：
  - 失去上游 bugfix/安全补丁的自动获得——需自己盯关键漏洞。
  - 砍企业级功能时可能触发隐藏耦合（如多租户渗透在数据层）——需评估剥离成本。
  - Fork 后的代码长期会与上游分叉，未来若想切回/换框架成本更高。
- **监控点**：若某框架的关键安全漏洞频发且自维护负担过重，
  应新写一条 ADR 重新评估（可能切到 F1 跟上游，或换框架）。

## 备选方案

- **F1. 长期跟上游 cherry-pick**：维护成本高，单人扛不住，放弃。
- **F3. 子集复制（抽目录）**：看似灵活，但"剥离"往往因依赖纠缠比 fork 整库更坑，
  且容易破坏框架的内在一致性，放弃。
- **F0. 继续 100% 自研 runtime**：已证明单人无法驾驭 Agent 运行时的全部复杂度，
  正是本次要解决的问题，放弃。
