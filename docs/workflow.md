# EvolveFlow 协作工作流

> 这份文档定义"人和 AI（ZCode）如何围绕这个项目长期协作"。
> 核心原则：**用 `/` 命令驱动，而非自然语言**——命令有结构、可重复、不会歧义。
> 核心原则：**先对齐，再执行**——任何重构/新功能前，先澄清意图。

---

## 一、技能（skill）使用工作流

技能通过 ZCode 的 `/` 菜单触发。下面按"何时用、解决什么、产出什么"列出。

### 地基层（项目知识骨架，建一次）

#### `/setup-matt-pocock-skills`

- **何时用**：项目首次规范化时；或要切换 issue 追踪方式时。
- **解决什么**：建立项目知识骨架——领域文档布局、issue 追踪、triage 标签。
- **产出**：`CONTEXT.md`、`docs/adr/`、`docs/agents/*.md`。
- **EvolveFlow 的推荐选择**：
  - Issue tracker → **local markdown**（单人项目，存 `.scratch/`）
  - 领域文档 → **single-context**（一个 CONTEXT.md 够用）
  - triage 标签 → 用默认值

#### `/ubiquitous-language`

- **何时用**：setup 之后；当对话中领域术语积累到一定量时。
- **解决什么**：消除"同一个词各说各话"——固化 capability、排程、锁定、dream 等术语。
- **产出**：`UBIQUITOUS_LANGUAGE.md`（术语表）。

### 需求层（每次有新想法时）

#### `/grill-me`

- **何时用**：冒出"我想加 X 功能"的模糊念头时。
- **解决什么**：无情拷问想法，逼你明确边界、验收标准、潜在风险。
- **产出**：被锤炼过的、经得起质疑的想法。

#### `/to-prd`

- **何时用**：grill 完、想法成熟后。
- **解决什么**：把含糊想法转成结构化产品需求文档。
- **产出**：一份 PRD（目标、用户故事、验收标准、非目标）。

### 维护层（隔很久回来时）

#### `/handoff`

- **何时用**：每次结束一个工作块、要离开项目一段时间前。
- **解决什么**：浓缩当前进度、未决问题、下一步，避免"回来重新考古"。
- **产出**：交接文档。

#### `/decision-mapping`（或手写 ADR）

- **何时用**：每次做了有取舍的技术决策（选型、架构、取舍）。
- **解决什么**：记录"当初为什么这么干"，半年后不抓瞎。
- **产出**：`docs/adr/NN-标题.md`。

## 二、推荐执行顺序（针对 EvolveFlow 现状）

1. **现在** → `/setup-matt-pocock-skills`（建地基）
2. **紧接** → `/ubiquitous-language`（固化语言）
3. **每次开发**：有想法 → `/grill-me` → `/to-prd` → 实现
4. **每次收工** → `/handoff`

---

## 三、代码改动的工作流

任何代码改动（修 bug、加功能、重构）遵循：

1. **先说意图**：在对话里讲清"要改什么、为什么"，不直接动手。
2. **对齐方案**：复杂改动先出方案让审批，简单改动直接说要做。
3. **小步验证**：每改一块就 `npm run build && npx vitest run && npm run lint`，不全绿不继续。
4. **更新文档**：代码改了，涉及的文档（ARCHITECTURE / CONTEXT / ADR）同步更新。
5. **记录决策**：重要取舍写一条 ADR。

### 质量门槛（合并前必须满足）

- `npm run build` 全栈构建通过
- `npx vitest run` 全测试通过
- `npm run lint` 0 errors（warnings 可接受）
- `npm run typecheck` 通过

## 四、文档维护规则

| 文档                     | 改动时机                                 |
| ------------------------ | ---------------------------------------- |
| `CONTEXT.md`             | 领域概念/边界变化时                      |
| `UBIQUITOUS_LANGUAGE.md` | 新术语出现或旧术语含义变化时             |
| `docs/adr/*.md`          | 做了新的重要技术决策时                   |
| `ARCHITECTURE.md`        | 架构/数据流/分层变化时                   |
| `README.md`              | 对外可见信息（功能、技术栈、用法）变化时 |
| `CHANGELOG.md`           | 每次发版                                 |

**铁律**：代码与文档对齐。文档说的必须和代码一致；不一致时，要么改代码要么改文档，不留糊涂账。
